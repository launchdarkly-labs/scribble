Scribble

A local-first way to annotate any HTML document and collaborate with agents on it.

Initial design notes · draft 0 ·

The problem

Agents are getting good at producing rich HTML artifacts — reports, dashboards, design docs — and HTML carries far more context than Markdown. But reviewing long agent-generated HTML hits a familiar wall: there is no good way to point at a paragraph and say "this number looks wrong" or "expand this" without copy-pasting fragments back into chat.

Tools like Hunk, Codiff, and Dunk have solved this for code diffs: a local daemon holds review state, a UI lets the human leave inline notes, and the agent reads and responds via CLI. We want the same loop, but for arbitrary HTML.

The shape of the thing

A small CLI, scribble, that:

  1. Opens any local HTML file in a browser overlay where the user can select text and leave comments.
  2. Stores annotations in a sidecar JSONL file next to the document (git-friendly, agent-readable).
  3. Exposes a CLI / HTTP surface so agents can list unresolved annotations, reply, and resolve them.
  4. Re-anchors annotations across document edits using W3C selectors.

Anchoring: the hard part is solved

Pointing at a span of rendered HTML in a way that survives edits is the central technical risk. The W3C Web Annotation Data Model already specifies a layered selector strategy:

  • TextQuoteSelector — exact text + small prefix/suffix snippets. Survives most edits.
  • TextPositionSelector — character offsets into the normalized text. Fast fallback.
  • RangeSelector — start/end XPath + offset. Precise but fragile.

The reference implementations are apache/incubator-annotator and the anchoring code inside hypothesis/client. Both are MIT/Apache and we can borrow shamelessly.

Open question. Existing libraries work but feel like 2014. The overlay UI, the selection affordances, the styling — everything looks dated. We probably want to keep the anchoring math from prior art and build a modern UI from scratch.

Annotation shape

{
  "id": "ann_01H...",
  "target": {
    "source": "report.html",
    "selector": [
      { "type": "TextQuoteSelector",
        "exact": "Q3 revenue grew 12%",
        "prefix": "Highlights: ",
        "suffix": " year-over-year" },
      { "type": "TextPositionSelector", "start": 4821, "end": 4840 }
    ]
  },
  "body": { "type": "TextualBody", "value": "is this number right?" },
  "author": "human",
  "status": "open",
  "replies": [],
  "created": "2026-05-19T10:14:00Z"
}

CLI surface (sketch)

scribble open ./report.html              # start daemon, open browser
scribble session list                    # active sessions
scribble list --unresolved --json        # annotations for an agent to address
scribble get <id>                        # full annotation + quoted text
scribble resolve <id> --reply "..."      # close with a response the user sees
scribble comment add --doc report.html \ # agent asks the human a question
    --quote "..." --summary "..."
scribble watch                           # stream changes (SSE)

JSON everywhere. Batch resolve apply --stdin for multi-comment replies. Session selection by --doc <path>, mirroring Hunk's --repo.

Three collaboration flows

A · Agent-initiated

Agent generates a long HTML doc, opens a session, asks the user to annotate, then polls until all annotations are resolved.

B · User-initiated

User opens a doc in scribble, annotates, then asks an agent: "address my comments." Agent runs scribble session list and processes them.

C · Mid-task check-in

During a long edit, the agent pins a question to a specific span ("should this section stay?") and waits for a targeted human reply. This is the sneaky-powerful one.

The skill

A scribble-review skill is as important as the CLI itself. It teaches agents:

  • When to open a session (long HTML, ambiguous review) vs. just talk in chat.
  • How to poll, batch, reply with substance instead of just resolving.
  • What not to do: don't edit sections with unresolved annotations pointing into them; re-anchoring may fail.

Write the skill first as the design doc — every awkward instruction is a sign the CLI shape is wrong.

v0 scope

  1. Daemon: serves doc in iframe, overlay UI in parent frame.
  2. Selection → comment, persisted to .scribble/<doc>.jsonl.
  3. CLI: open, list, get, resolve, session list.
  4. Re-anchor on read using TextQuoteSelector with prefix/suffix.
  5. Skill file, ~80 lines.

Out of scope for v0: multi-user, auth, remote sync, rich-body replies, edits-from-annotations.

Open questions discuss

  • One doc per daemon, or multi-doc sessions like Hunk?
  • Re-anchor eagerly on file change or lazily on read?
  • Agent replies as threaded notes, or just status + reply field?
  • Annotation UI: what does a modern overlay look like? Margin notes? Floating bubbles? Linear sidebar? Linear-style command bar? This is the next thing to noodle on.
